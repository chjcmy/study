# SQL vs NoSQL

---

## 비교

| 특징 | SQL (RDBMS) | NoSQL |
|------|-------------|-------|
| **구조** | 정해진 스키마 (Table) | 유연한 스키마 (Document, Key-Value) |
| **관계** | JOIN으로 연결 | 보통 JOIN 없음 (임베딩 or 별도 조회) |
| **확장성** | 수직 확장 (Scale-up) 위주 | 수평 확장 (Scale-out) 용이 |
| **트랜잭션** | **ACID** 엄격 보장 | **BASE** (유연한 일관성) |
| **예시** | MySQL, PostgreSQL | MongoDB, Redis, Cassandra |

---

## 언제 무엇을 쓰는가?

### SQL이 좋은 경우
1.  **데이터 구조가 명확하고 변경이 적음**
2.  **관계가 복잡함** (다대다 관계가 많음)
3.  **데이터 무결성이 중요** (금융, 결제)
4.  **JOIN이 필수적일 때**
    *   예: 사용자-주문-상품-배송 연결 조회

### NoSQL이 좋은 경우
1.  **데이터 구조가 자주 변경됨** (유연성 필요)
2.  **읽기/쓰기 성능이 매우 중요** (대규모 트래픽)
3.  **수평 확장이 필요** (데이터가 페타바이트급)
4.  **데이터 간 관계가 단순** (독립적 문서)
    *   예: 로그 데이터, 실시간 채팅, 제품 카탈로그

---

## NoSQL의 종류

### 1. Document (문서형)
*   **MongoDB**, CouchDB
*   JSON 유사 형식으로 데이터 저장
*   스키마가 유연하여 개발 속도 빠름
*   용도: 콘텐츠 관리, 카탈로그, 일반적인 웹 앱

### 2. Key-Value (키-값)
*   **Redis**, DynamoDB
*   단순하고 엄청나게 빠름 (O(1))
*   복잡한 쿼리는 어려움
*   용도: 캐싱, 세션 저장, 장바구니

### 3. Column-Family (컬럼형)
*   **Cassandra**, HBase
*   대량의 쓰기 작업에 최적화
*   컬럼 단위로 저장하여 분석에 유리
*   용도: 시계열 데이터, IoT 센서 로그

### 4. Graph (그래프)
*   **Neo4j**
*   데이터 간의 "관계" 자체가 중요할 때
*   용도: SNS 친구 추천, 사기 탐지, 경로 찾기

## 면접 핵심 포인트

```
Q: SQL과 NoSQL의 가장 큰 차이는?
A: 스키마 유무와 확장성 방식.
   SQL은 엄격한 스키마와 수직 확장이 특징이고(데이터 무결성 유리),
   NoSQL은 유연한 스키마와 수평 확장이 특징입니다(대규모 트래픽 유리).

Q: RDBMS도 수평 확장이 가능한가?
A: 샤딩(Sharding)을 통해 가능하지만, 관리가 복잡하고 JOIN 제약이 생김.
   NoSQL은 태생적으로 수평 확장을 고려해 설계되어 훨씬 용이함.

Q: 스타트업 초기에는 무엇을 추천?
A: 데이터 구조가 자주 바뀌면 NoSQL(MongoDB)이 빠를 수 있지만,
   복잡한 쿼리가 필요하고 안정성이 중요하면 RDBMS(PostgreSQL)가 무난.
   최근엔 PostgreSQL도 JSON 타입을 지원하여 유연성을 가짐.
```
