# 코틀린의 모듈 (Module)
#Kotlin/Concept

> [!INFO] 모듈이란?
> **모듈(Module)**은 코틀린 프로젝트를 구성하는 독립적인 배포 단위입니다. 모듈은 함께 컴파일되는 코틀린 소스 파일들의 집합으로, 일반적으로 IntelliJ, Android Studio, Gradle 등에서 하나의 모듈은 다음과 같은 것들을 의미합니다.
> - Gradle 소스 셋 (e.g., `main`, `test`)
> - IntelliJ IDEA 모듈
> - Maven 프로젝트

모듈은 큰 프로젝트를 여러 개의 논리적 단위로 분리하여 코드의 재사용성, 유지보수성, 그리고 빌드 속도를 향상시키는 데 중요한 역할을 합니다.

## 1. 모듈과 `internal` 가시성 제한자

코틀린의 가시성 제한자 중 **`internal`**은 모듈과 직접적인 관련이 있습니다.

> [!IMPORTANT] `internal`
> `internal`로 선언된 클래스, 함수, 프로퍼티는 **선언된 모듈 내부에서만** 접근할 수 있습니다. 다른 모듈에서는 이들을 참조할 수 없습니다.

이는 라이브러리나 프레임워크를 개발할 때 매우 유용합니다.
-   **`public`**: 외부 모듈에 공개하여 누구나 사용할 수 있도록 할 API
-   **`internal`**: API를 구현하기 위한 내부적인 로직이지만, 외부에는 노출하고 싶지 않은 코드

```kotlin
// my-library 모듈
package com.example.library

// 외부 모듈에 공개할 API
public class LibraryApi {
    fun process() {
        // 모듈 내부에서만 사용 가능한 Helper 호출
        val helper = InternalHelper()
        helper.doSomething()
    }
}

// my-library 모듈 내부에서만 사용 가능
internal class InternalHelper {
    fun doSomething() {
        println("Internal logic is running...")
    }
}
```

```kotlin
// app 모듈 (my-library 모듈에 의존)
package com.example.app

import com.example.library.LibraryApi
// import com.example.library.InternalHelper // 컴파일 오류! internal 클래스는 import 불가

fun main() {
    val api = LibraryApi()
    api.process() // OK
}
```

## 2. 모듈의 장점

-   **정보 은닉**: `internal`을 통해 모듈의 내부 구현을 외부로부터 완벽하게 숨길 수 있습니다.
-   **느슨한 결합 (Loose Coupling)**: 모듈들은 서로의 공개된 API에만 의존하므로, 한 모듈의 내부 구현 변경이 다른 모듈에 미치는 영향을 최소화할 수 있습니다.
-   **빌드 시간 단축**: 멀티모듈 프로젝트에서는 변경된 모듈과 그에 의존하는 모듈만 재빌드되므로, 전체 프로젝트의 빌드 시간을 단축할 수 있습니다.
-   **협업 효율 증가**: 팀 단위 개발 시, 각 팀이 담당하는 모듈에만 집중하여 작업할 수 있어 충돌을 줄이고 생산성을 높일 수 있습니다.

## 3. 모듈과 패키지의 차이

| 구분 | 모듈 (Module) | 패키지 (Package) |
| :--- | :--- | :--- |
| **개념** | 함께 빌드되고 배포되는 코드의 **물리적 단위** | 코드의 네임스페이스를 구성하는 **논리적 그룹** |
| **역할** | 의존성 관리, 빌드 단위, `internal` 가시성 범위 | 이름 충돌 방지, 코드 구조화 |
| **관계** | 하나의 모듈은 여러 개의 패키지를 포함할 수 있음 | 하나의 패키지는 여러 파일에 걸쳐 존재할 수 있음 |
